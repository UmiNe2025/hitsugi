# M24 戦闘表示・ダンジョンマップ全面改善指示 — Claude/Fable 5向け

作成日: 2026-07-12  
対象: `Battle.tsx` / `Dungeon.tsx` / PixiJS探索描画  
実装担当: Claude/Fable 5  
正典: `docs/GDD_v3.md` §8.9

## 0. 結論

今回の刷新は、エフェクトや装飾の追加ではなく、次の二点を中心に行う。

1. **戦闘**: 誰が誰へ何をするのかを、人物の大きさ・距離・灯脈で一瞬に読めるようにする。
2. **探索**: 主人公、歩ける場所、危険、次に調べたいものを、暗さを保ったまま判別できるようにする。

共通の署名要素は **「灯路 / 灯脈」** とする。探索では移動先まで短く灯る道、戦闘では行動者と対象を結ぶ一撃の脈として使う。常時光らせず、判断または行動の瞬間だけ現れる。

## 1. 現状確認

### 1.1 実機確認

2026-07-12、ローカルdev版を1280×720で新規開始し、出立から「宵の森 地下1層」まで確認した。

- HUD、加護選択、地域導入、地域別色調は実装済み。
- ダンジョンは36×22タイルに対し、横1280pxでは約35タイル相当が表示され、フロアの大半が一度に見える。
- 主人公と特殊物が小さく、背景・壁・敵影の暗色が近いため、探索対象より広い暗色面が先に見える。
- マップ外または奥壁の純黒に近い面が画面下部を大きく占める場合がある。
- ログ、隊員HP、D-pad、ミニマップ、資源、階層名が四隅へ分散している。
- 戦闘は行動順、火脈、ログ帯、コマンド盤、隊員札、人物名札がすべて別の小UIとして存在する。
- 味方立ち姿は約80px、敵は約112pxを基準としており、全画面戦場に対して人物が小さい。
- 攻撃の踏み込みは約30pxだが敵味方間は数百pxあり、攻撃者と被弾者が接触せず「別々の絵が揺れる」印象になりやすい。

### 1.2 変更してはいけないもの

- 戦闘計算、技威力、敵AI、報酬、灯消費は視覚Phaseへ混ぜない。
- `maps.gen.ts` を手編集しない。地形変更は生成器とシードから行う。
- UI効果音を個別buttonへ重複配線しない。
- M23の `RegionVisualProfile`、四幕、痕跡、ランドマークを削除せず土台として使う。
- 既存の背景、敵絵、ボス絵、歩行spriteを捨てない。

## 2. ビジュアル方針

### 2.1 主題

対象は、家族の命を背負って暗い夜藪へ入り、灯を頼りに生還する和風世代交代RPG。画面の仕事は「豪華に見せる」ことではなく、家族が危険へ踏み込んでいる実感と、次の判断を同時に伝えること。

### 2.2 色

既存トークンを維持する。

| 役割 | 色 | 用途 |
|---|---:|---|
| 墨夜 | `#060810` | 画面外、最深部 |
| 藍帳 | `#10172E` | UI面、情報の土台 |
| 古金 | `#C9A86A` | 見出し、確定、地名 |
| 命火 | `#FF9D45` | 現在手番、選択、灯路 |
| 傷朱 | `#C73E3A` | 被弾、危機、敵意 |
| 和紙 | `#EFE6D4` | 本文、最重要数値 |

金と橙を常時発光させない。命火は「いま操作するもの」または「一撃の瞬間」に限定する。

### 2.3 文字

- 地名/主名/勝鬨: Shippori Mincho 700、24〜42px、字間0.18〜0.35em。
- 操作/本文: Shippori Mincho 500〜600、13.5〜16px。
- HP/MP/倍率/順番: 数字のみtabular-nums。12.5px未満にしない。
- 長い説明を戦場へ置かない。戦場上は一文20文字程度を上限にする。

## 3. 戦闘「灯脈の間」

### 3.1 PCレイアウト

720px高を次の三段へ固定する。`min-height: 60vh` と下部UIの積み上げに依存しない。

```text
┌ 第3巡  [今 燈吾] [餓鬼] [小鈴] [濡れ女] ───────────────┐ 52px
├──────────────────────────────────────────────────────┤
│  敵の兆し                                                地名 │
│                                                          │
│     [敵1 160px]    ── 灯脈 ──       [味方1 160px]          │
│        [敵2]            火脈2連            [味方2]          │
│                                                          │
│                 直近の戦況 一行                           │
├──────────────────────────────────────────────────────┤
│ [手番人物 HP/MP/状態] [攻撃][技][防御][逃走] [技/対象の説明] │ 152px
└──────────────────────────────────────────────────────┘
```

- `.battle-screen` を `grid-template-rows: 52px minmax(0, 1fr) 152px` 相当にする。
- 戦場は画面高の65〜72%を占める。
- 通常敵/味方は140〜180px。1体時は180〜220px。ボスは220〜300px。
- 敵味方の足元を共通の地平線へ揃え、接地影と背景の地面パースを一致させる。
- 1〜4体で配置テンプレートを分ける。人数を式だけで散らして人物が端へ逃げないようにする。
- UIと戦場を一緒に揺らさない。被弾時は背景/人物の内側stageだけを揺らす。

### 3.2 人物の配置

- 敵陣の占有域: 画面幅18〜43%。
- 味方陣の占有域: 57〜82%。
- 主対象同士は画面中央に近づけ、非対象は奥へ0.86〜0.94倍で退く。
- 行動中のみカメラ相当のstage transformを1.04〜1.08倍へ寄せる。
- 死者の空スロットを残さず、生存者を再配置する。ただし位置が飛ばないよう300msで補間する。

### 3.3 一撃の流れ

攻撃アニメは距離を無視した30px移動をやめ、対象との位置差から接触点を求める。

1. 溜め 100〜140ms
2. 対象方向へ接近 160〜220ms
3. 接触点で45〜70msのhit-stop
4. 被弾、数字、属性片、SEを同じフレームへ集約
5. 180〜240msで元位置へ戻る

接触時の攻撃者と対象の見た目上の間隔は12px以内。遠距離技は接近せず、灯脈を弾道として対象へ届かせる。

### 灯脈

- 行動者の足元から対象の足元へ、細い命火色の線が150msで走る。
- 通常攻撃は一度、連撃は脈動回数、回復は命火から和紙色へ変える。
- 命中後300ms以内に消す。
- 火脈連数/倍率は中央の常設丸印ではなく、灯脈上へ一時表示する。
- reduced-motionでは線を静的に150〜250ms表示し、拡大/揺れを止める。

### 3.4 情報を統合する

- 味方HP/MPを人物名札と下部隊員札へ二重表示しない。
- 通常時は人物足元に名前＋HP。現在手番だけ下部にHP/MP/状態/技を詳しく表示する。
- 敵名札は名前、HP、属性、状態だけ。相性は対象選択時に `有利/不利/等倍` を文字付きで示す。
- 直近ログは戦場下端に一行だけ。全履歴は「戦況の記」で開く。
- `chain` など実装語をプレイヤー向け文言へ出さない。「攻撃＝継続」などの開発用ヒントは手引きへ移す。
- 開幕の戦利品スロットは戦況判断を遮るため、報酬予告へ名前を変え、右上へ小さく置くか勝利時へ統合する。

### 3.5 コマンド盤

- 左: 現在手番の人物肖像/名前/HP/MP/状態。
- 中央: 攻撃、技、防御、退く。主要4操作を1列または2×2で固定。
- 右: 選択中の技の威力、属性、範囲、消費、対象相性。
- 技選択でコマンド盤全体を縦長一覧へ置換しない。中央に技一覧、右に説明を維持する。
- 敵を直接押した通常攻撃と「攻撃」ボタンの挙動は同じにする。
- 対象選択中は全対象へ44px以上の選択領域と番号を表示し、1〜4キーでも選べるようにする。

### 3.6 敵の兆し（第二Phase）

視覚刷新と分離して実装する。

- 正確な技名やダメージを先に公開せず、「前衛を狙う」「全体へ溜める」「守りを崩す」程度を示す。
- 行動順上の敵chipに `兆` 印を出し、対象候補の足元へ薄い傷朱の波紋を出す。
- AIが行動時に再抽選する現仕様なら、先に行動計画を確定する設計変更が必要。別commit・別テストにする。

### 3.7 ボス戦

- 既存のボス背景/カットインを使う。
- 開幕は「地域景→主の輪郭→主名→戦場」の1.4〜1.8秒。操作可能になるまで2秒を超えない。
- ボスを通常敵と同じ112px枠へ入れない。
- HPを画面上部の主専用ゲージへ分離し、段階変化/危険技を同じ場所へ示す。
- 勝利時は紙吹雪より先に、ボスの灯が消える、地域色が戻る、一族が呼吸を整える順で見せる。

### 3.8 モバイル

```text
┌ 行動順 横スクロールなし ┐
│          敵             │
│       灯脈 / 戦況        │
│          味方            │
├ 手番人物 HP MP 状態 ─────┤
│ [攻撃] [技] [防御] [退く] │
└────────────────────────┘
```

- 390×844で戦場45〜55%、コマンド盤を下部固定。
- 技一覧/説明はbottom sheet。戻るを常時表示する。
- 横持ちではPC三段構造を縮小して使う。
- 重要情報をPCだけに置かない。

## 4. ダンジョン「灯路の夜藪」

### 4.1 カメラ倍率を最優先で直す

現行は1280px幅で約35タイルが見え、主人公が小さすぎる。TILE定数を戻して地形生成を壊すのではなく、描画worldへレスポンシブzoomを導入する。

- PC横画面: 横22〜26タイル、縦13〜17タイルを目安にする。
- モバイル縦画面: 横10〜12タイルを目安にする。
- PCの主人公表示高48〜64px、モバイル38〜52px。
- zoom目安は1.25〜1.50。実値は1280×720、1440×900、390×844で決める。
- カメラ追従は即時固定でなく、80〜120msの軽い追従遅延を付ける。
- 進行方向へ0.8〜1.5タイル先を見せるlook-aheadを付ける。
- マップ端でも画面の半分が純黒にならないよう、screen固定の地域別backdrop/霧層で外側を埋める。

### 4.2 床・壁・障害物を三段階で読む

- 歩行床: 現状より明度を12〜20上げ、局所的な材質模様を加える。
- 接触壁: 床との境界へ2〜3pxの縁光と落影を付ける。
- 奥壁/画面外: 暗いまま維持し、プロップを置きすぎない。
- 壁セル全面を同じ矩形に見せず、連続壁を大きな塊として描く。
- 木、岩、竹、墓石がすべて黒い障害物へ見えないよう、形状、縁色、足元影を地域別に分ける。
- 主人公の足元へ小さな命火の接地円を置き、床から浮かせない。
- 主人公より手前の高いプロップはalpha 0.35〜0.55へ退避し、通過後に戻す。

### 4.3 灯路 — タップ移動の手応え

タップBFSは村エンジン(`src/village/engine.ts`)のみに実装済み。ダンジョン(`src/dungeon/engine.ts`)には未実装のため、村エンジンのBFS/path構造を移植した上で入力結果を視覚化する(2026-07-12訂正: 旧記述「タップBFSは実装済み」はダンジョン側には当てはまらなかった)。

- タップ地点へ命火色の目的印を100ms以内に表示する。
- 発見済み床だけ、先頭4〜7マス分の細い灯路を表示する。
- 歩くたび後ろから消す。
- 到達不能地点は傷朱の割れ印を出し、無反応にしない。
- 手動移動へ切り替えた瞬間に灯路を消す。
- reduced-motionでは点線を静的表示する。

### 4.4 敵影

- 敵影の輪郭を背景プロップと区別する。属性色を全面塗りせず、目、縁、足元だけへ使う。
- 通常時は1pxの冷たい縁、警戒時は傷朱の足元波紋と頭上 `!`。
- 追跡開始時に一瞬だけ主人公方向へ細い視線を出す。
- 金の敵影は形状または粒子でも区別し、色だけに頼らない。
- 5タイル以内の敵影は、眷属なしでも背景へ完全に埋没しない。

### 4.5 特殊物と目的

- 宝箱、焚火、祠、石碑、階段を現状の1.25〜1.6倍へ。
- 発見時に一度だけ縦の淡光または音を出す。常時点滅は禁止。
- 画面上部の地名plateへ「踏査」だけでなく現在の短期目的を一つ表示する。
  - 例: `石碑を探す 0/2`
  - 階段発見後: `下り階段を見つけた`
  - 灯15%未満: `帰り火を考えよ`
- 方向矢印は発見済み対象だけに出す。未知の階段位置を漏らさない。

### 4.6 HUD再配置

```text
┌ 灯78 / 宵の森 B2 / 石碑1/2 ─────── [小休止] ┐
│                                              │
│                  探索画面                     │
│                                    [地図]     │
│                                              │
│ [隊 HP]    一行の出来事              [D-pad]  │
└──────────────────────────────────────────────┘
```

- 灯、地名、階層、短期目的を左上〜上中央の一つの帯へ統合する。
- 資源は常時最重要ではないため、右上の小さな回収札へまとめる。
- ミニマップはPC 140〜170px、モバイル88〜110px。拡大表示を切り替えられるようにする。
- 隊員HPは左下の短い横札。1人でも横へ伸ばさない。
- 長い3行ログを常設しない。直近一文を主人公上または画面下中央へ3〜5秒出し、履歴は小休止から読む。
- D-padはタッチ端末だけ強く表示し、マウス/キーボード環境ではopacityを下げるか隠す。

### 4.7 地域差を色替えで終わらせない

各実装済み地域は、次の5層のうち最低3層を固有化する。

1. 地面材質: 土、苔、板、石、骨、灰、水膜
2. 境界形状: 木立、竹垣、石壁、墓列、柱、崖
3. 署名物: 地域固有ランドマーク2種以上
4. 空気: 蛍、雨、灰、霧、花粉、星屑
5. 音: 風、水滴、鈴、虫、軋み

同じ4基盤themeを残してよいが、静止画を白黒にしても二地域を区別できる形状差を一つ以上入れる。

### 4.8 フロアの物語

M23の四幕を画面変化へ強く反映する。

- 閾: 入口が背後に見え、地域署名物を一度見せる。
- 行: 部屋ごとに小さな発見。目的物の密度を均等散布しない。
- 畏: 彩度と環境音を少し落とし、主の痕跡を増やす。
- 座: 通常プロップを減らし、広い余白と主背景で対峙を作る。
- 鎮: 討伐後は金色粒子だけでなく、敵密度、環境音、壊れた物の復元で変化を示す。

### 4.9 地形生成の次段階

描画改善後に実施する。先にマップ全生成をやり直さない。

- 部屋archetypeを5種用意: 広間、回廊、中庭、橋、段丘風。
- 各フロアへ「記憶に残る一室」を一つ保証する。
- 通路幅2〜3は維持しつつ、入口から最初の判断地点までを短くする。
- 行き止まりには宝、石碑、景観のいずれかを置き、空振りを減らす。
- 生成後検証へ画面上の障害物密度、長い直線、同形部屋反復を追加する。

## 7. 受入条件

### 7.1 戦闘

- 1280×720で戦場が画面高65%以上、コマンド/人物/ログの重なり0。
- 味方1/4人、敵1/2/4体、ボス1体の全組合せで人物が切れない。
- 通常1体時の人物高が140px以上。ボスが通常敵より明確に大きい。
- 近接攻撃のimpact時、攻撃者と対象の見た目上の距離12px以内。
- 被弾時にHUD/コマンド盤は揺れない。
- HP、現在手番、選択対象、技消費、相性を500ms以内に特定できる。
- 390×844で主要4コマンドがスクロールなしで押せる。
- Tab、Enter、Space、1〜4キー、Escapeで対象選択と取消ができる。
- reduced-motionで情報欠落がない。

### 7.2 ダンジョン

- 1280×720で横22〜26タイル、390×844で横10〜12タイルを基準にする。
- PCの主人公表示高48px以上。
- マップ端で説明のない純黒面が画面の25%以上を占めない。
- 歩行床、接触壁、奥壁を静止画で判別できる。
- 敵影を5タイル以内で背景プロップと識別できる。
- タップ後100ms以内に目的印、250ms以内に最初の移動反応を返す。
- 到達不能タップが無反応にならない。
- 宝、焚火、祠、石碑、階段を3秒以内に視認できる。

- PCで平均55fps以上、標準的なモバイルで平均30fps以上。新規全画面filterは最大2pass。
- D-pad、帰り火、ミニマップ、sheetがsafe-areaと重ならない。
