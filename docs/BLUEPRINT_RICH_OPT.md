## 1. エグゼクティブサマリ

- **ゴール**: ライブ稼働中の『灯継ぎ』を、体験品質(リッチ)と基盤品質(最適化)の両面で一段引き上げる。
- **最上位軸**: 不可逆損失(プレイヤーのセーブ資産・配信可用性)の防衛を可逆な成長(コンテンツ追加)より先に置く。
- **本書の性格**(ゲート指摘の明記): これは「リッチ化コンテンツ計画」ではなく**基盤ハードニング計画**である。リッチ化コンテンツ側は [GENRE_GAP_ANALYSIS.md](GENRE_GAP_ANALYSIS.md)(状態異常/難易度/デイリー等・要ユーザー在席)が正本であり、**基盤収束後の次サイクルで成長次元を明示的に立てる**ことを残条件とする。
- **主要設計判断3つ**:
  1. **D1 配信**: sharp+mozjpeg一括再圧縮(拡張子維持=コード無変更)を第一手に。[事実(実測)] q75で31%削減・画像はPages上で無圧縮素通し配信=削減が転送量に直結。
  2. **D2 品質ゲート**: CIに tsc+oxlint を追加(迂回路=workflow_dispatch維持)+vitest(node環境)で純関数コア+**旧版実セーブJSONのfixture回帰を最優先**。[事実(コード)] 現CIはbuild→deployのみで型エラーでもデプロイされ得る・テスト0。
  3. **D3 セーブ堅牢性**: 実装順=chronicle有界化(kind優先の予算制)→世代交代時ローテバックアップ→検証つきフォールバック→**QuotaExceededError握り潰しの解消**。[事実(コード)] save.ts:11-13の空catchが最大欠陥。

## 3. 決定的次元の具体化

### D1: 配信性能 — 推奨: mozjpeg一括再圧縮+サムネ生成(scene-lazyは後段)
- **[事実(実測)]** 画像2107枚/199MB(>250KB=111枚、150-250KB=379枚)。sharp+mozjpeg q75で**31%削減**(サンプル5枚: 1572→1382KB)。WebP q80は28%でフォールバック要→**mozjpegが優位**(拡張子維持=コード無変更・互換100%)。
- **[事実(実測)]** GDI+再圧縮はq72でも12%(既存パイプラインでは絞れない)。Pages配信: 画像は素通し(458KB実測)・JSはgzip 481KB・max-age=600。
- **動線記録**: `npm i -D sharp` → node実測(5枚×3方式)/ `curl -sI -H "Accept-Encoding: br,gzip"` ×3種。
- 推奨手順: ①`scripts/recompress-images.mjs`(sharp+mozjpeg q75、ポートレート系(face_/god_/boss_)はq82、原本をassets_src/origへ退避)②サンプル100枚バッチ→**目視スポットチェック**(バンディング確認)→全量 ③Expeditionサムネ用`_thumb`生成(76×44表示に350KB原本使用中)。
- 代替: WebP/AVIF(`<picture>`フォールバック要・iOS15以下切捨て)— 追加30%が必要になった時の第二段。
- scene-lazy(Dungeon/Battle/Pixi層のdynamic import)は初回JS短縮の本筋だが改修範囲が広い→後段(構築順序⑤)。
- 撤退分岐: 目視で画質劣化が許容外→q82全量 or 対象カテゴリ除外。

### D2: 品質ゲート — 推奨: CI 3行+vitest(node)+セーブfixture最優先
- **[事実(コード)]** CI=build→deployのみ(型チェックなし)。テスト0。コア(battle 379行/inheritance/save/epitaph)は純関数中心。
- 推奨: ①CIに`npx tsc -b && npx oxlint`追加+`workflow_dispatch`迂回路維持 ②vitest(**node環境・jsdom不要**=純ロジックのみ)③**旧版実セーブJSON fixtureの回帰テスト**が最優先(唯一の不可逆損失=プレイヤー資産を守る。v1/v3/v4各形式を固定資産化)④battle(elementMult/chain/crit)/inheritance(遺伝率)/save(migration)の20-40件。
- 運用規律(risk-narumi条件): テストは「壊れたら痛い純関数だけ」・カバレッジ数値は追わない・false red 3連続なら削除・テストは実プレイ確認の代替ではなく追加。
- 代替: UIテスト(@testing-library)→見送り(1人体制で最も腐りやすい)。

### D3: セーブ堅牢性 — 推奨: 有界化→ローテ→フォールバック→quota解消(この順)
- **[事実(コード)]** save.ts:11-13が空catch(quota超過を無言で握り潰し=オートセーブが黙って失敗し進行消失し得る)。chronicle/familyは無限成長。localStorage≈5MB[事実(MDN)]。
- 推奨(architect指摘の依存順): ①chronicle有界化 — **kind優先の予算制**(death/birth/era等は保持、event/dailyを古い順に間引く。一律カウント切りは初期世代史を失い主題的コスト大)②世代交代時にbak1-3ローテーション ③読込時main→bak1→bak2→bak3の**検証つき**フォールバック(family配列+seasonIndexの妥当性チェック)④quota検知→UI警告+剪定再試行(握り潰し解消)。
- B2連動: セーブに`lastExportAt`を記録し、タイトルで「最終バックアップからN日」リマインド。
- 代替: IndexedDB→保留(saveGameが同期・store密結合のためasync化連鎖が過大。多スロット/>5MB要件化時に再検討)。

## 5. 横断チェック
- **コスト**: 全構築0円(sharpはdevDependency導入済)。任意: カスタムドメイン~1.5k円/年(B2緩和・ユーザー判断)。※cfo-yuki相場感でなく実額—Pages/GoatCounter/Sentry無料枠は現条件であり恒久保証はない。
- **スケール**: 現アーキで月間~5,000セッションまで安全圏[推定(帯域100GB÷15MB/セッション)]。超過見込みでCloudflare Pages移行(B2前提)。
- **人的資源**: 1人+AI。テスト・画像パイプラインとも維持可能規模に絞った(削除規律つき)。
- **法規制/データ所在**: 個人データ非収集・Cookie不使用・セーブはユーザー端末内。※単一視点・要専門確認(公開拡大時)。
- **ドキュメント/DX**: WORKLOG/GDD/STATUS/GENRE_GAP/ASSET_NEEDS/本書で網羅。store.ts 1902行の分割は次サイクルのDX課題。
- **主要リスク**: ①B1×B2×D3連鎖(帯域cliff→移行→オリジン変更→セーブ全喪失)— 本書の中核発見。D3+B2緩和を移行より先に実装することで遮断 ②mozjpeg画質劣化(目視工程で防ぐ)③CI固着(actionsバージョンピン+迂回路)。

## 6. 構築順序(クリティカルパス)
1. **D2-①** CIにtsc+oxlint(3行・迂回路つき)— 最安で最大の回帰防止
2. **D2-③④** vitest導入+セーブfixture回帰+コア純関数テスト — D3実装の安全網を先に張る
3. **D3-①〜④** chronicle有界化→ローテバックアップ→フォールバック→quota解消(+B2リマインド)
4. **D1-①②③** mozjpegサンプル100枚→目視→全量再圧縮+_thumbサムネ
5. **D1-scene-lazy**(任意・大改修)Dungeon/Battle/Pixi層のdynamic import
6. **B1** GoatCounterアカウント有効化(ユーザー作業)→帯域監視の実効化
- 最初に潰す不確実性: **mozjpeg全量での実削減率と画質**(サンプル100枚バッチ+目視=最安検証)。
