# MISSION M30: ダンジョン非描画(空キャンバス)の調査 — STATE

作成: 2026-07-17。/mission 起動(ユーザー報告=ダンジョンに入ると真っ暗・HUDのみ表示)。

## ①契約
**ゴール**: 通常プレイでダンジョンのPixiJSキャンバスが描画されない不具合を再現・原因特定・修正し実機で実証。
**監査区分**: 自己監査。**再現→修正→再現消滅**を確認するまで完了としない。

## ②調査結果(証跡)
- 通常プレイ経路 `departDungeon('mushishigure_michi')`+boonDraft を dev で再現テスト:
  - 単発: 加護解消後の暗部率 **2.1%**(=完全描画)。テストフック経路(2.0%)と同等。
  - **6周回の入退場サイクル**: 全て暗部率2-3%・WebGLコンテキスト喪失なし・engine非破棄・console error 0。
- `build`緑 / playwright gate+baseline(ダンジョン暗部率・HUD)**12 passed** / 既存190も緑。
- CSS差分(M28-D)はHP札のみで`.dungeon-canvas`不変。init()ガード(M29)は engDestroyed=false=不発火。
- **結論: dev/build/gate では正しく描画。ユーザー報告の空キャンバスは再現せず=環境要因が濃厚**
  (デプロイ直後のブラウザキャッシュ / WebGL一時不全 / GPUドライバ)。アプリ自体は起動(HUD描画)、
  キャンバスのみ空=バンドル404ではなくPixiJS描画層の問題。

## ③適用した対応
- **描画初期化失敗の可視化**(`Dungeon.tsx`/`dungeon_m25.css`, commit): `engine.init()` を `.catch` で捕捉し、
  WebGL不可/素材失敗で例外時は「再読み込み/郷へ戻る」オーバーレイを出す(従来は void で沈黙→空画面)。
  ※init()が例外を投げる系の失敗を可視化。例外を伴わない描画不全は別途(context-loss recovery が候補)。

## ④保留(ユーザー側情報が必要)
- **ハードリロード(Ctrl+Shift+R)で解消するか** → 解消なら原因=キャッシュ。
- 解消しない場合の**コンソール(F12)の赤エラー** → 原因の確定に必要。
- WebGLコンテキスト喪失recovery(webglcontextlost/restored)の追加は、上記診断後に要否判断。
