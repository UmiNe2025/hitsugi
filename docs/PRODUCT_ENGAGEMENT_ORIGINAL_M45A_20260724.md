## 1. 結論

M45の診断と8施策は妥当である(現行コードとの照合で「現行実装から見える空白」表の全行を裏取りした。§9に検証方法を記す)。追補すべきは施策の数ではなく、**M45の8施策がすべてテキスト・scene・UIパネルの次元に属している**という一点の偏りである。

「選択→記憶→再会→次代で回収」というM45の核ループを支持した上で、本書は**記憶が住む場所を画面テキストの外へ広げる**3次元+共通基盤1件を追補する。

1. **空間の記憶** — 郷の歩行マップとダンジョンのフロアに、一族の歴史が物として残る(O-01星送りの灯 / O-04野辺の墓標)。
2. **象徴の記憶** — タイトル画面が「何代続いた家か」を一目で映す(O-03星還りの空)。O-01と対をなす: 郷の灯は星へ**送った**証、タイトルの星は**還った**先 — 同じ宇宙観の両端であり、死者数の再掲ではない(§5d)。
3. **感覚の記憶** — 血脈に音の主題が宿り、継承・看取りで再帰する(O-05血脈の音)。
4. **共通基盤** — M45 P0-01の「約束の反響」を一般化した**反響チャネル**(O-00)。以後のすべての「事実一致の一行再掲」がここを通り、§6の月次予算をコードで一元強制する。

いずれも新規の物語文章をほぼ持たず(全施策合計で新規短文≤12、既存資産の再掲が主)、戦力報酬ゼロ、強制sceneゼロである。

## 3. 共通基盤 O-00「反響チャネル」

**これは施策ではなく、P0-01実装時に汎用として作る配送機構である。**

M45 P0-01は「生涯途中の反響は同一月最大1件、1場面10秒以内」を施策内ルールとして定めた。O-02(辞世)・O-04(墓標)・P1-03(宿敵の痕跡)も同型の「事実一致の非modal一行」であり、各施策が独自に頻度制御を持つと§6の「同一月の強制scene最大1件」「deferred queue純増0」を**合算で**破る抜け道が生まれる。

**最小実装**:

- `pushEcho(source, text)`一関数。source enum = `vow | epitaph | grave | nemesis`。
- **予算の二層構造**(vowの飢餓防止): `vow`はチャネルの予算を消費しない**別格**であり、P0-01自身のルール(生涯三回・同一月最大1件)だけに従う — M45の主役が他sourceに食われることは構造上ない。チャネル予算は**vow以外の合算で同一月最大1件**。
- 逐次発火の挙動を確定する: 月内**先着1件が予算を消費**し、後着は捨てる(繰り越さない=queue純増0の機械保証)。「優先順位」は**同一処理内で複数sourceが同時発火した時だけ**の裁定規則: grave > epitaph > nemesis。
- 表示は既存の非modal一行枠(帰還ログ/Home帯のいずれか既存様式)を再利用し、新しい表示コンポーネントを作らない。
- 同一内容の再掲防止は`flags`動的キー(`echo_<source>_<id>`)で行う。前例: `flags['mikiri_'+regionId]`(store.ts:1918-1924)、セーブスキーマ変更なし。

**§6適合**: sceneを作らない・queueを持たない(捨てる)・報酬なし。予算の一元化により、以後どの反響施策が増えても§6違反が構造的に起きない。

**測定**: チャネル自体はevent `echo_shown`(enum source)のみ。P0-01の`vow_echo`はこのチャネルの第一乗客として発火する(M45 §4のkey名と互換)。

## 4. 追補施策

### O-01 星送りの灯 — 郷に灯る一族の記憶

**体験**: 郷の歩行マップの片隅に、逝った者の数だけ小さな献灯が並ぶ。星神の世界観では死者は星へ還る — 郷の灯はその**送りの証**である。夜藪から帰るたび、家が積み重ねてきた喪失と継承が**歩ける風景**として目に入る。説明もUIもない。ただ灯が増えていく。

**最小実装**:

- `resolveVillageVisualState()`(src/core/data/village_visual_state.ts:35-66)の出力に`fallenCount: number`を1フィールド追加する。入力Pickには`family`が既に含まれる(同:26-29)ため、入力変更なし・純関数のまま。
- 郷エンジンの`buildProps()`(src/village/engine.ts:1017)に献灯スポット固定列(`KIN_SPOTS`と同形式、engine.ts:105参照)を追加し、`min(fallenCount, 12)`基だけ`addProp`(engine.ts:1027)+`glowAt`(engine.ts:712-737、breathe周期は祠と同じ1500/0.10)で置く。両APIはシグネチャ変更不要。
- 12基超は最奥の1基だけを強め、輝度は`min(1, log2(fallenCount+1)/log2(13))`で漸増させる(数字は出さない)。reduceMotionは既存の構築時スナップショット方式に従う。
- 新規文章0。タップ・当たり判定なし(非対話プロップ=既存BFS/衝突/テストに無影響)。
- 命名ノート: ゲーム内の既存機構「帰り火」(緊急帰還)と紛れないため「送り火」の語は避け「星送りの灯」とする。実装前にGDD_v3の世界観語彙(常夜/灯/星神)と表記を照合すること。

**機構的差分**: M45の全施策と異なり、読む行為を要求しない**常設の環境ストック**。P2-01(家宝額装)が「選んで飾るUI」なのに対し、星送りの灯は選択なし・自動蓄積・空間配置。

**測定**: 観察票のみ(初見8名中、献灯の意味を問われて「死んだ家族の数」と答えられる人数)。event追加なし。

**見送り条件**: 説明できる人数3/8未満なら演出強化ではなく撤去する(郷の視覚ノイズを増やさない)。郷の描画負荷でモバイルfps liveness(fps>1)を割る場合は基数を6へ半減。

**§6適合**: scene/queue/報酬/save変更すべてなし。M33で確立した郷Tier1技術(既存addProp/glowAt)の範囲内であり、却下済みの郷Tier2(生成アート・NPC・マップ拡張)に踏み込まない。

### O-02 辞世の反響 — 1370首を場所と季節に返す

**体験**: 先代が果てた地域へ子が初めて足を踏み入れた時、または命日の季に郷で月を送った時、その者の辞世が一行だけ、静かに再掲される。「──露と落ち 露と消えにし 我が身かな(◯◯・第3代)」。新しい文章は一切書かない。**プレイヤーが看取りの場で一度読んだ句が、意味を変えて帰ってくる**。

**最小実装**:

- 発火条件の入力: 既存フィールド`Character.epitaph / deathSeason / deathCause`(types.ts:149-153)+現在の`seasonIndex`、および**O-04前段でcaptureする新フィールド`deathRegionId`**(条件aはO-04前段の完了に依存する — 独立ではない)。
  - 条件a: 出立先regionId === 先代のdeathRegionId(初回のみ)
  - 条件b: seasonIndex % 12 === deathSeason % 12(命日の季、故人1名につき一度のみ)
- 表示はO-00チャネル経由(source=`epitaph`)。月1予算・再掲防止・非modalはチャネルが保証。
- 新規文章0(枠文「──」+句+名+代のみのテンプレ1種)。

**機構的差分**: P0-02「記憶の糸」は関係**事実**をscene内で呼び戻す(新規文章を生成)。本施策は**既存の詩そのもの**を、場所・暦の一致で非modal再掲する。生成なし・scene外・対象は故人全員(一組限定でない)。

**測定**: `epitaph_echo`(enum: region|anniversary)。§4様式の表は§6に併記。

**見送り条件**: 表示後の観察で「誰の句か分からない」が5/8を超える場合、条件aのみに絞る(命日はコンテキストが弱い可能性)。

**§6適合**: scene/queue/報酬なし。saveは`flags`動的キーのみ(チャネル共通)。辞世の**追加量産はしない**(§7凍結を遵守 — 再利用のみ)。

### O-03 星還りの空 — タイトルに積もる家の歴史

**体験**: タイトル画面の夜空に、これまで逝った一族の数だけ小さな星が灯る(四十八を超えた家は、個々の星ではなく天の川の霞が濃くなってゆく)。周回(NG+)しても星は消えない。**起動した瞬間に「自分は何十人を看取ってきた家の当主だ」と分かる**。ゲームを消しても家は消えていない、という約束の画面化。

**最小実装**:

- `TitleScreen`(src/ui/Title.tsx)で既存`exportableSaveData()`(src/core/save.ts:446)を呼ぶ(現在は`inspectSaveSlot`の状態文字列しか読んでいない — Title.tsx:16)。星数 = `family.filter(!alive).length + (flags.ancestorStars ?? 0)`。
- NG+持ち越し: `newLegacyGame()`(store.ts:735-764)で`flags.ancestorStars += 今周の逝去者数`を加算。前例: 同関数が`flags.ngCycle`を既に持ち越している。動的flagsキーのためセーブスキーマ変更なし。
- 描画はDOM/CSSの微光ドット(決定論的配置=indexシード)。**表示星数は`min(count, 48)`で打ち止め、超過は星でなく「天の川霞」1レイヤの濃さで表す**: `opacity = min(0.3, 0.05 × floor(超過数/12))` — 個々の星を数えさせず、長く遊んだ家ほど空全体が仄かに満ちる(飽和の逆転を霞で回避)。操作なし・ボタンなし・タイトル一枚絵の背後レイヤ。reduce-motionでは瞬きを止め静止表示。
- セーブなし(none)の時は星0(現行と同一の画面)。

**機構的差分**: P1-04「前回の灯」は6時間条件つきの再開**情報**(前回・今・次+CTA)。本施策は条件なし常設の**象徴**で、情報量を持たず(数字も名前も出さない)、操作もない。両者は同居可能で干渉しない。

**測定**: 観察票のみ(再訪8名中、タイトルの星の意味を説明できる人数)。event追加なし(タイトルは計測前のためlocal metricsの起点より前に位置する)。

**見送り条件**: M42が「壊さず伸ばすもの」と定めたタイトルの明快さ(一枚絵・余白・明快な入口)を損なう — 具体的には初見観察で「入口が分かりにくくなった」指摘が1件でも出る — 場合は輝度を下げ、それでも出る場合は撤去する。

**§6適合**: scene/queue/報酬なし。save変更はflags動的キー1個のみ(migration不要・冪等)。

### O-04 野辺の墓標 — 倒れた地を家が覚えている

**体験**: 全滅で失った先代が倒れた地域に、次の代が挑む。出立画面の地域札に「この地で◯◯が果てた」と一行。その階に降りると、入口近くに墓標がひとつ増えている。**宿敵(P1-03)が敵側の因縁なら、これは身内側の因縁**である。取り返す動機を、敵ではなく場所が担う。

**最小実装(2段構え — captureを先行)**:

- **前段(capture・表示なし)**: `Character`に`deathRegionId?: string`と`deathFloor?: number`をoptional追加。書き込みは死亡3経路のうち位置が存在する2経路のみ:
  - 歩行ダンジョン全滅(store.ts:2447-2449) → `run.regionId`+`run.floor`。`run`は同アクション冒頭の`const run = get().dungeonRun`(store.ts:2220)で取得済みのスコープ内変数。
  - 旧ノード探索全滅(store.ts:2631-2639) → **`exp.regionId`のみ**(`const exp = d.expedition` store.ts:2495、`exp.regionId`参照の先例は同:2498。floor概念が存在しないため書かない)。
  - 寿命死(store.ts:456) → 書かない(郷で逝く者の記憶はO-01星送りの灯が担う — 空間の記憶の分業)。
  - セーブ作法はM26以降の確立手順に従う: optional追加+isValidSave(save.ts:156-225)へ型ガード追加。migrate関数は不要(欠損=郷死として自然に成立)。
- **後段(表示)**: 出立画面の地域札に一行(既存region-card、非modal)。歩行ダンジョンで`run.regionId+floor`が一致する階に入った時、**入口3タイル以内の固定スポット**に墓標1基+献灯色の微光1点を置き、**接近時はO-00チャネル(source=`grave`)経由で一行を出す** — 既存の墓所クラスタ(props.ts:394-414はzaka/tani系で25%確率・4〜7基をランダム生成する)に埋没するため、視覚差分だけに意味を負わせない。チャネル予算で表示されなかった月は再掲防止flagを立てず、後日の接近で再発火する(queueでなく再試行=純増0のまま、O-00の先着制と整合)。floor不明(旧探索死)の地域は出立札の一行のみ。
- **複数回全滅の扱い**: 地域あたり墓標は上限1基とし、最新の死者で更新する。出立札の一行も最新1名のみ。**累積数は一切出さない**(数えさせない=責めに見せない)。
- **意匠のトーンガイドライン**: 墓標は糾弾的でない意匠(苔むした自然石調・血痕/敗北記号なし・色調は地域テーマに同化)。敗北の証ではなく「家が覚えている」しるしとして描く。
- 新規短文はテンプレ2種以内。

**機構的差分**: P1-03(宿敵)は**敵**の生存・強化・再戦。本施策は**身内の喪失の座標**で、戦闘に一切影響しない。captureの前段は「現在store.ts:2465の年代記文生成で使った直後に位置情報を捨てている」データ損失の是正であり、後段のすべての表示施策(および将来の家系図拡張)の前提になる。

**測定**: `grave_echo`(表示時)+capture健全性は機械test(決定論fixtureで3経路の書込を検証: 歩行全滅=region+floor / 旧探索全滅=regionのみ / 寿命死=undefined)。

**見送り条件(事前+事後の二段)**: ①**pilot前の内部レビュー**で意匠・文言をトーンガイドラインに照合し、「責めに見える」アラートが1件でも出た時点で墓標部分を凍結し出立札の一行のみへ縮退する(§6の未達罰禁止は事前審査で担保し、実プレイヤーを検出器にしない)。②pilot後も「責めに見える」観察が2/8を超えれば同縮退。

**§6適合**: scene/queue/報酬なし。save変更はoptional 2フィールド(冪等・破損時は欠損=郷死へ自然縮退)+isValidSaveガード。**captureは今この瞬間も失われ続けているデータの保全であり、表示施策と独立にPhase 0へ置く価値がある**。

### O-05 血脈の音 — 世代モチーフ

**体験**: 継承の場で、その家系に固有の短い三音が鳴る。以後、血の呼応(血縁連携)と看取りの場で同じ三音が形を変えて帰ってくる。言葉にされない「家の音」。

**最小実装**:

- 既存プロシージャル音源(src/core/audio.ts — 平調子`HIRAJOSHI`+`deg()`:10-15、`pluck`:142/`bellTone`:245等)に`motif(seed)`公開メソッドを1つ追加。`se()`のunion(audio.ts:365-368、現21種)は変更不要。
- **写像式を確定する**(実装者に音楽判断を求めない): `seed = 家祖idの文字コード総和`。3音は平調子の度数index `[seed % 5, (seed >> 2) % 5, (seed >> 4) % 5]`を`deg()`に渡して得る。音色は`pluck`固定・全長約1.2秒・音量は既存SEと同係数。
- 発火3箇所: 継承scene表示時 / 血の呼応発動時 / 看取りscene表示時。いずれも既存イベントに音を1回添えるだけで、新規演出枠を作らない。
- 音量・消音・reduce系は既存設定(マスター音量/消音)に自動的に従う(audio既存経路)。

**機構的差分**: M45の8施策に聴覚は一切ない。先例はM期wave Gの戦闘SE差別化(会心=太鼓強打+高箏、`critHit`/`weakHit` — audio.ts:376-381)で、「ゲームイベントに意味の音を与える」路線の血脈版。UI効果音の自動付与規約(`attachUiClickSfx`)はボタン押下音の話であり、ゲームイベント音の個別実装は既に確立された別系統。

**測定**: 観察票のみ(一世代5名中、「継承の音に気づいた」人数)。音のevent計測は行わない(§4の値制約に音は馴染まない)。

**見送り条件**: 気づいた人数0/5なら三音の設計を変えず**撤去**する(気づかれない音はコストではないが、拡張もしない)。「不快・怖い」が1件でも出れば即撤去。

**§6適合**: scene/queue/報酬/saveすべてなし。
